# Deployment

How the site reaches the world. The land it sits on, the path from commit to visible page, the commitments each part of that path must honor. This file exists because "where does the site actually live" is a decision the code alone cannot answer — the same build output can land on ten different hosts and the site's posture (privacy, performance, resilience, cost) shifts with each.

Deployment is a grounds concern. The house stands on the grounds; the grounds are tended regardless of whether anything in the house changes. When something about the deployment model changes (a new host, a new CDN rule, a new environment), this file is where the reason is named before the config moves.

---

## What Gets Deployed

The entire deployable surface is the contents of `dist/client/` after `pnpm build`:

- `index.html` and per-route `{room}/index.html` — prerendered HTML for every known path (see `RENDERING_STRATEGY.md`).
- `assets/*.js` — the hydration bundle and per-route chunks (~118KB gzipped main chunk).
- `assets/*.css` — the design-system stylesheet, self-hosted fonts' `@font-face` rules included.
- `assets/*.woff2` — Literata and Newsreader, per-subset (see `PRIVACY.md`'s Fonts section).
- `favicon.svg`, `apple-touch-icon.svg` — from `public/`.
- `robots.txt` — from `public/`.
- `sitemap.xml` — generated by TanStack Start at build time from the prerender set.
- `pages.json` — the sitemap's companion index.

`dist/` itself also contains an SPA fallback shell (`dist/index.html`) and a `dist/server/` directory used during prerender. Neither is deployed. The root `dist/index.html` would serve any route not prerendered; for a pure-static build with all routes enumerated, it is unused by visitors and should not override per-route HTML on any host.

---

## Deployment Target

**Pure static.** No Node runtime is required at request time. `dist/client/` can be served by any host that speaks HTTP, sets correct MIME types for `.js`, `.css`, `.woff2`, `.svg`, `.xml`, and `.json`, and respects `index.html` as a directory default.

This is a deliberate constraint. The `RENDERING_STRATEGY.md` holds the Fuller Horizon — server functions, API routes, streaming SSR — any of which would require a runtime. Adopting any of them changes this file's "what gets deployed" section and the host's capability requirements. Until then, the static constraint unlocks the full set of static hosts.

## Host Requirements

Any host under consideration must satisfy:

- **HTTPS by default.** TLS on every route; automatic certificate renewal. No "add HTTPS later."
- **Custom domain support.** The site lives at `danielbdyer.com` (named as the canonical URL in `src/shared/seo/schema-org.ts` and in the sitemap host). The host must support a CNAME or A/AAAA record to a stable endpoint.
- **SPA-free routing.** The host must serve `dist/client/{path}/index.html` for the URL `/{path}/` and return 404 (or fall back to the prerendered `/` only if explicitly configured) for paths that do not exist. Generic "SPA mode" that returns `/index.html` for every 404 would defeat the prerender — visitors would see the foyer at every wrong URL with a 200 status, which is wrong for SEO and wrong for the reader.
- **Preview deployments.** Every pull request builds and deploys to a unique preview URL. The preview is where hydration, motion, theme toggle, and reveal animations can be exercised in a real browser before merge — the repository has no browser-based automated test for post-hydration behavior, so the preview is the checkpoint.
- **Privacy posture that aligns with `PRIVACY.md`.** The host's access logs are the one place a visitor's IP and user-agent are recorded by any party other than the visitor's ISP. Hosts that do not retain identifiable IPs, or that let the site owner control retention and access, are preferred. Hosts that merge access logs into their own analytics product, or that sell or share them, are declined.
- **No required third-party CDN or script injection.** The site does not include any tracker, analytics tag, or chat widget. A host that inserts its own script into the served HTML (some do by default) must support disabling that behavior before it is acceptable.

## Candidates

Specific hosts are not chosen here; the choice is `DEPLOYMENT.md`'s when the first deploy is taken up. Names are for orientation:

- **Cloudflare Pages.** Strong privacy default (no IP retention by default; access-log retention configurable), global edge network, preview deployments, HTTPS automatic. Build and CDN are Cloudflare's own; no third-party in the request path.
- **GitHub Pages.** Simplest from this repository's perspective — lives in the same organization as the source. HTTPS automatic. No preview deployments on PR without extra tooling; access logs not easily auditable.
- **Netlify.** Mature, preview deployments per PR, HTTPS automatic. Access logs and analytics are Netlify's; privacy requires reading their current terms carefully.
- **Vercel.** Similar surface area to Netlify. Historically more aggressive about analytics integration; requires more care to align with `PRIVACY.md`.
- **Self-hosted (VPS, Fly.io, etc.).** Full control, full responsibility. Reasonable only if a clear operational commitment is there.

Given `PRIVACY.md`'s posture, Cloudflare Pages and GitHub Pages are the leading candidates. The actual choice is Danny's.

## DNS and SSL

- **Apex domain:** `danielbdyer.com` points at the chosen host. The specific record type (A/AAAA flattened, ALIAS, ANAME) depends on the host.
- **WWW:** `www.danielbdyer.com` redirects 301 → `danielbdyer.com` (or vice versa — the canonical choice is captured in `src/shared/seo/schema-org.ts`'s `SITE_URL`, currently the apex).
- **SSL:** automatic via the host. The site is not a certificate authority and does not manage its own keys.
- **Caching at the DNS layer:** TTLs are whatever the host recommends; no manual tuning until a concrete reason names itself.

## CI/CD

`.github/workflows/ci.yml` already runs typecheck, eslint, vitest, `pnpm audit`, `pnpm build`, and Lighthouse on every push to main and every PR. Deployment extends that workflow:

- **On every PR:** the workflow builds, runs all checks, and triggers a preview deploy. The preview URL is commented on the PR.
- **On merge to main:** the workflow builds, runs all checks, and deploys to production. The previous production build remains reachable (as the current live version) until the new build is verified by the host; rollback is a redeploy of the previous commit's artifact.
- **On tag (optional, future):** if release tags are adopted, a tagged deploy can be published to a durable URL for archaeology.

No `vercel.json`, `netlify.toml`, or similar host-specific config lives in the repo today. When a host is chosen, its config file lands alongside this one as the authoritative declaration.

---

## Environments

Two environments, no more until something else earns one:

- **Preview:** per-PR URL. Ephemeral — destroyed when the PR closes. Carries the same build output as production would. Used to verify hydration, motion, theme, and reveal manually in a browser before merge.
- **Production:** `danielbdyer.com`. The public site.

A staging environment is not held. The preview-per-PR flow already gives the value a staging environment would give (a place to verify changes before they go live) without adding a separate long-lived URL.

## Caching

Static assets fingerprinted by Vite (`index-Hash.js`, `index-Hash.css`, font woff2 files with content hashes) can be cached aggressively:

- **Fingerprinted assets (`/assets/*`):** `Cache-Control: public, max-age=31536000, immutable`. The file's content defines its URL; if the content changes, the URL changes.
- **HTML (`*.html`):** `Cache-Control: public, max-age=0, must-revalidate`. HTML references asset URLs by hash; stale HTML would reference old assets. The client should always re-check HTML freshness.
- **`robots.txt`, `sitemap.xml`, `pages.json`:** `Cache-Control: public, max-age=3600`. An hour is fine; these change rarely and a brief delay between publish and crawler re-fetch is not a problem.
- **Fonts (`*.woff2`):** inherit the fingerprinted-asset policy.

The exact syntax for setting these headers lives in whatever host config file the chosen host requires (a `_headers` file on Cloudflare Pages / Netlify; a `staticwebapp.config.json` for Azure Static Web Apps; etc.). The rules above are the authoritative source for those files.

---

## Rollback

Every production deploy is addressable by the commit SHA that produced it. Rollback is a redeploy of a previous commit — no database to migrate, no server to restart, no state to reconcile. The host's "promote previous deploy" button, or a manual `git revert` followed by a push, both work.

Because deploys are idempotent over the content of `dist/client/`, there is no risk of partial rollback. Either the new HTML is live or the old HTML is live.

## Web Vitals Forwarding

`src/shared/seo/web-vitals.ts` is wired and logs metrics to the console in development. Production forwarding to a privacy-respecting analytics provider is held until a provider is chosen; see `PRIVACY.md` for the commitments any forwarding must honor (aggregate only, no visitor identifier, no IP retention, no cross-site correlation). When a provider is chosen, the forwarding wire-up lands alongside the host choice — both are "before first deploy" decisions.

## Held

Each of these can be named here without being committed. When it's time to make the commitment, the item moves from Held to an earlier section.

- **A `_headers` or equivalent file.** Concrete cache-control headers per path. Triggered by the host choice — the exact file format depends on the host.
- **A `_redirects` file.** Currently no redirects are needed. An apex/www canonical choice will probably produce one rule.
- **Error pages hosted at `/404.html`.** The prerender already produces per-route `index.html`s; a 404 page served by the host for unknown paths would avoid the fallback-to-SPA antipattern named in Host Requirements.
- **Bot policy beyond `robots.txt`.** If spam or scraping becomes an issue, a host-level rate limit or bot-mode rule may be named. Not today.
- **Multi-region deployment.** The site is globally small; a single edge network is enough. Multi-region earns a name only when one region's performance is measurably worse than another.
- **CI release notes.** When tagged releases become a pattern, automatic release-notes generation from the git log can be considered. For a personal site with continuous publishing, tags may never become the primary cadence.
